规约实践 - IEC 61850概念（重点）

导入依赖：

<dependency>
    <groupId>org.openmuc</groupId>
    <artifactId>openiec61850</artifactId>
    <version>1.7.0</version>
</dependency>

规约实现分为客户端和服务端，在实际生产中，我们一般扮演的都是客户端向服务端发起请求的角色，在此之前我们先了解一些规约实践需要了解的一些概要：

Data Model - 数据模型

IEC 61850 服务器包含一个树状数据模型，该模型的叶子包含可由客户端访问的数据（整数、布尔值、字符串等）。客户端还可以从服务器检索整个数据模型。

最上面的模型节点称为“服务器”。在 IEC61850bean 中，它是 ServerModel 类型的对象。服务器节点包含 1..n 个逻辑设备 (LD)。一个逻辑设备可以包含 1..n 个逻辑节点 (LN)。一个逻辑节点可以包含 1..n 个数据对象。在 IEC61850bean 中，逻辑节点（LN）不包含完整的数据对象，而是包含所谓的功能约束数据对象 (FCDO)。FCDO 是按功能约束分割的数据对象。FCDO可以包含其他FCDO、数组、构造的数据属性和/或基本数据属性的组合。

IEC61850bean 中服务器模型的所有节点都属于以下七种类型之一：

  • ServerModel（服务器模型）

  • LogicalDevice（LD，逻辑设备）

  • LogicalNode

  • FcDataObject

  • Array

  • ConstructedDataAttribute

  • BasicDataAttribute

它们都实现了 ModelNode（直译过来就是模型节点，也就是我们需要获取的数据节点） 接口。节点 FcDataObject、Array、ConstructedDataAttribute 和 BasicDataAttribute 也实现称为 FcModelNode 的接口，因为它们在标准中被视为功能约束数据。IEC 61850 的许多服务只能应用于功能约束数据（例如GetDataValues 和SetDataValues）。

在对客户端进行编程时，您可以通过来自服务器设备的 ClientAssociation.retrieveModel 或通过来自 SCL 文件的 SclParser.parse 获得服务器模型的副本（这个后面会提及，这个很重要）。使用第二种方法时（即通过SCL文件获取副本），必须使用 ClientAssociation.setServerModel 在客户端关联中设置模型。对服务器进行编程时，您可以通过 ServerSap.getModelCopy() 函数获取服务器模型的副本。

然后，您可以使用多个函数浏览模型：

  • ServerModel.findModelNode(ObjectReference objectReference, Fc fc) 将搜索具有给定引用和功能约束的子节点。

  • ModelNode.getChild(String name, Fc fc) 将返回具有给定名称和功能约束的子节点。

  • ModelNode.getBasicDataAttributes() 将返回模型节点的所有叶子节点（基本数据属性）的列表。

补充一点：模型副本其实就是刚刚提到的树状数据结构，我们需要提前获取到模型副本后，才能直到服务端能提供什么数据

 

Client - 客户端

如果您想要连接到 IEC 61850 服务器，您应该首先创建 ClientSap（SAP = 服务访问点）实例并根据您的需要对其进行配置。然后使用 Associate() 方法建立与服务器的关联

xxxxxxxxxx
ClientSap clientSap = new ClientSap();
association = clientSap.associate(remoteAddress, remotePort,
                    authenticationParameter, localAdress, localPort, eventListener);


Runtime.getRuntime().addShutdownHook(new Thread() {
            @Override
            public void run() {
                association.close();
            }
        });

remoteAddress与remotePort：即你要连接的远程主机IP、Port；

authenticationParameter：一般为null即可

localAdress, localPort：为你本地连接的IP与Port。

注意：这一般填的不是你电脑的ip，而是服务商给你的白名单IP，你只有使用该IP才能连接到服务端。Port的值一般填0，填其他的端口可能会有冲突

eventListener：事件监听器

此时客户端就可正常连接上，并在需要的时候自动关闭了

 

Server - 服务端

首先使用 ServerSap.getSapsFromSclFile() 方法获取 ServerSaps 列表。此方法从给定的 ICD 文件读取 SAP。获取您想要运行的 ServerSap 并根据您的需要对其进行配置（例如设置要侦听的端口）。ServerSap 包含 ICD 文件中定义的完整设备模型。使用 getModelCopy() 方法检索它的副本。使用 startListening() 告诉 ServerSap 开始侦听配置的端口。这是一个非阻塞函数。

这个是用来自己本地模拟服务端的情况用的：

xxxxxxxxxx
serverSap = new ServerSap(port, 0, null, SclParser.parse(sclFilePath).get(0), null);
serverSap.setPort(port);
serverSap.startListening(this);
serversServerModel = serverSap.getModelCopy(); // 获取模型副本

这种是来让你本地模拟服务端与客户端交互的情况的，加深你自己对其的了解，这里只简单提示一下。你可以到GitHub官网下载源码，里面有提供相关样例

实际生产中，服务端不需我们提供，我们只是作为客户端与其交互而已

DataSet - 数据集

数据集（DS）形成一组/一组数据。数据集可用于使用单个请求/响应消息交换一次读取或写入多个数据对象/数据属性。用于此目的的服务是 GetDataSetValues 和 SetDataSetValues。除了数据集服务之外，报告、日志记录、GOOSE 和采样值服务也使用数据集概念。

属于数据集一部分的数据属性或数据对象称为数据集的成员。只有功能约束数据才可以是数据集的成员。

IEC 61850 标准定义了五种与数据集相关的 ASCI 服务：

  • GetDataSetValues获取数据集所有成员的值。

  • SetDataSetValues设置数据集所有成员的值。

  • CreateDataSet动态创建新的数据集。

  • DeleteDataSet删除已创建的数据集。

  • GetDataSetDirectory获取服务器中所有现有数据集的列表

存在两种类型的数据集：

  • 持久数据集- 它们具有 LDName/LNName.DataSetName 格式的引用。它们对所有客户都是可见的。持久数据集可以在SCL文件中预先配置。在这种情况下，它们无法被删除。持久数据集也可以由客户端动态创建。在这种情况下，它们稍后可能会再次被删除。一旦服务器停止，动态创建的数据集将被自动删除。

  • 非持久数据集- 它们具有格式 @datasetname 的引用。它们仅对通过 CreateDataSet 服务创建它的客户端可见。这些数据集仅在协会开放时存在。

Report - 报告（重点）

写在前面，设置报告可以实现服务端数据的自动响应，即客户端扮演等待数据更新的角色，而非主动请求数据的一方

报告允许服务器根据事件发送数据，而无需客户端明确请求。发送哪些数据以及导致报告的事件是通过所谓的报告控制块 (RCB) 进行配置的。

该标准区分两种类型的报告：缓冲报告和非缓冲报告。使用缓冲报告，服务器会缓冲报告，以防与客户端的连接中断。这样，可以在客户端再次连接后发送报告。缓冲报告是通过缓冲报告控制块（BRCB）配置的。无缓冲报告是通过无缓冲报告控制块 (URCB) 配置的。

RCB 位于服务器数据模型（即服务器的ICD/SCL 文件）的逻辑节点内。它们是固定的，无法在运行时删除或添加。一旦客户端注册接收报告（通过启用它），RCB 就会被阻止。这意味着单个 RCB 无法并行向多个客户端发送报告。所以，在代码实践中，如若设置不当，直接enableReport的话，服务端会响应有其他服务占用含义的错误。

RCB 始终与特定数据集相关联，该数据集必须与 RCB 位于同一逻辑节点中。如果发生任何配置的事件，将报告数据集成员的子集。RCB 的关联数据集可以由客户端在运行时动态更改。

RCB 位于服务器模型树中的逻辑节点中。因此，RCB 的引用采用 LogicalDeviceName/LogicalNodeName.RCBName 的形式。尽管RCB只能引用与RCB本身位于同一逻辑节点中的数据集，但它们仍然可以监视来自其他逻辑节点的数据，因为数据集的成员可以位于其他逻辑节点中。

以下事件可能导致服务器发送报告：

  • 客户发出一般性询问。

  • 作为关联数据集一部分的数据属性发生更改或更新。

  • 通过设置 RCB 的完整性周期，可以定期发送报告。

这些事件之一是否确实导致发送报告是通过 RCB 配置的。

Report control block（报告控制块）包含以下字段：

  • 配置字段。仅当 RCB 尚未启用或被其他客户端保留时才能写入它们：

    • RptID - 报告 ID 标识 RCB。如果未设置，则等于 RCB 参考。RptID 将随每个报告一起发送，以便客户可以识别负责该报告的 RCB。

    • DatSet - 与此 RCB 关联的数据集的引用。它必须与 RCB 位于同一逻辑节点。每当配置的事件发生时，就会报告该数据集的成员。RCB 中设置的数据集引用必须包含美元符号而不是点，以将逻辑节点与数据集名称分隔开，例如：“LDevice1/LNode$DataSetName”

    • OptFlds - 位串，其中每个位指示可选字段是否包含在由该 RCB 引起的报告中。存在以下可选字段：序列号、报告时间戳、包含原因、数据集名称、数据引用、缓冲区溢出、条目 ID、分段和配置修订。对于 URCB，缓冲区溢出和 EntryID 的值将被忽略。

    • BufTm - 如果发生导致报告的事件，服务器将等待BufTm毫秒来处理其他事件。由于该时间跨度内的事件而要报告的所有数据都在单个报告中发送。

    • TrgOps - 指定哪些事件将触发报告。可能发生的事件有：

      • 数据更改 (dchg)

      • 质量变化（qchg）

      • 数据更新（dupd）

      • 完整性 - 如果启用，服务器将定期向客户端发送完整性报告。完整性报告将包含引用数据集所有成员的当前值。

      • 一般审讯（GI）。

    • IntgPd - 完整性周期指定定期发送完整性报告的间隔（以毫秒为单位）。

    • PurgeBuf（仅限 BRCB 的一部分，而非 URCB）-

    • EntryID（仅限 BRCB 的一部分，而非 URCB）-

  • 信息字段： 只能读取：

    • SqNum - RCB 的当前序列号。发送报告后，RCB 会将序列号加一。

    • ConfRev - 配置修订是一个计数器，表示引用的数据集更改的次数。

    • 所有者- 应等于保留 RCB 的客户端的 IP。如果 RCB 未保留，则应为 NULL（即空字符串）。

    • TimeOfEntry（仅限 BRCB 的一部分，不包括 URCB）

  • 功能字段：可以读取或写入 RCB 的以下字段来执行功能：

    • 保留（仅限 URCB 的一部分，而非 BRCB） - 在对 RCB 进行任何操作之前，客户端应通过将此字段设置为 true 来保留它。如果设置成功，RCB 将专门为此客户端保留。启用尚未保留的 RCB 会隐式保留它。

    • ResvTms（仅部分 BRCB，而非 URCB）- BRCB 的此属性是可选的。如果不存在，则通过简单地启用它来保留控制块。ResvTms 值为 -1 表示控制块是通过配置为一组特定客户端保留的。值为 0 表示 BRCB 未被保留。客户端可以通过写入大于0的值来保留控制块。该值表示在关联关闭或中断后保留应保持的秒数。

    • RptEna - 通过将此变量设置为 true，将启用报告。请注意，只要启用报告，更改/写入 RCB 和配置字段就会失败。

    • GI - 一般审讯。将此参数设置为 true 将启动向客户端发送报告。该报告将包含关联数据集的所有数据。发送报告后，服务器将自动将 GI 参数重置为 false。

报告始终包含以下信息字段：

  • RptID - 报告 ID 标识导致生成报告的 RCB。它等于 RCB 的 RptID 字段。

  • OptFlds - 等于相应 RCB 的 OptFlds 字段。

  • SqNum（可选，如果 OptFlds.SequenceNumber 为 true，则包含）- 报告的序列号。在发送时，它等于相应 RCB 的 SqNum 字段。

  • TimeOfEntry（可选，如果 OptFlds.report-timestamp 为 true，则包含）- 指定创建条目 ID 的时间。

  • DatSet（可选，如果 OptFlds.dataset-name 为 true，则包含）- 在此报告中发送其数据的数据集的引用。

  • BufOvfl（可选，仅当它是缓冲报告且 OptFlds.buffer-overflow 为 true 时才包含）- 指示服务器由于缓冲区溢出而丢失报告条目。仅适用于缓冲报告。

  • EntryID（可选，仅当它是缓冲报告且 OptFlds.entryID 为 true 时才包含） - 条目标识 ToDo

  • ConfRev（可选，仅当 OptFlds.conf-rev 为 true 时才包含） - 它等于相应 RCB 的 ConfRev 字段。

  • SubSqNum（可选，如果 OptFlds.segmentation 为 true，则包含在内）- 如果长报告不适合一条消息，则应将单个报告分为多个子报告。每个子报告应具有相同的序列号和唯一的 SubSqNum。

  • MoreSegmentsFollow（可选，如果 OptFlds.segmentation 为 true，则包含）- 指示后面有更多具有相同 SqNum 的子报表。

  • Inclusion Bitstring - 长度等于引用数据集的数据成员数量的位串。每个比特指示相应的数据成员是否包括在报告中。

  • Data-references （可选，如果 OptFlds.data-reference 为 true，则包含）- 报告的数据集成员的引用。

  • Values- 报告的数据集成员的值。

  • Reason Codes（可选，如果 OptFlds.reason-for-inclusion 为 true，则包含）- 原因代码指示每个报告值的报告原因。可能的原因包括数据更改、质量更改、数据更新、完整性、一般询问和应用程序触发。对于单个报告值来说，可能存在多种原因。

想要从某个 RCB 接收报告的客户端通常会首先尝试保留该 RCB。如果成功，他将按照自己的意愿配置 RCB。最后他将启用报告功能。
